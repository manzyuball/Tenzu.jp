# Tenzu.jp 大規模リワーク実行計画（日本語版Wikipedia準拠）

## 0. 目的
Tenzu.jp を「現在の日本語版Wikipedia」の体験に可能な限り近づけるため、UI・文体・情報設計・メディア・運用プロセスを含む超大規模リワークを半自走で進める。

## 1. 基本方針
- **合法性・倫理性を担保**しつつ、Wikipedia風の体験を設計する（完全な複製ではなく、互換性重視）。
- 実装は **反復開発（スプリント）** で行い、都度ユーザー確認を挟む。
- 進捗・品質・承認状態を `docs/wikipedia-rework-tasks.json` で一元管理する。

## 2. 完了条件（Definition of Done）
以下をすべて満たした時点で完了とする。

1. **UI一致度**: 主要テンプレート（トップ・記事・一覧・404・ナビ）が定義済み比較項目で「AまたはB評価」かつ平均 85%以上。
2. **文体一致度**: 記事テンプレートの文体ルール適合率が 90%以上。
3. **情報設計一致度**: 見出し階層、脚注、関連リンク、カテゴリ導線が定義済み要件を満たす。
4. **デザイン一致度**: 配色・余白・タイポグラフィ・コンポーネントの監査項目で重大差分 0件。
5. **メディア運用確立**: 画像生成ガイドライン、出力フォーマット、差し替え手順、ライセンス注記が整備済み。
6. **運用自走性**: タスク更新フロー、承認ゲート、回帰チェック、リリース手順が文書化済み。
7. **ユーザー満足**: 最終レビューでユーザーから明示的な承認（OK）を取得。

---

## 3. 具体的なフェーズ計画（6週間想定）

### Phase 0（Day 1-2）現状計測と基準化
- UI差分監査表を作成（ヘッダ/サイドバー/本文/脚注/リンク状態）。
- タイポグラフィ・余白・色の現状値を抽出。
- 記事本文の文体サンプルを採取し、スタイル逸脱を分類。

**成果物**
- `docs/ui-gap-audit.md`
- `docs/style-baseline.md`

### Phase 1（Week 1）デザインシステム整備
- Wikipedia風トークン（色・文字サイズ・余白・罫線）を策定。
- `assets/css/wiki.css` と `assets/styles.css` の責務分離。
- 共通レイアウト（ヘッダ、左カラム、本文コンテナ、脚注）の骨格を統一。

**成果物**
- `docs/design-tokens.md`
- ベースCSS改修PR

### Phase 2（Week 2）記事ページUI統一
- 見出し、目次、注釈、脚注、出典、テンプレート（infobox/hatnote/ambox）を統一。
- 記事ページ3本を先行改修（代表ページを固定）。
- PC/タブレット/モバイルで崩れ検証。

**成果物**
- 改修済み記事3ページ
- UI比較スクリーンショット

### Phase 3（Week 3）情報設計とナビゲーション再構築
- カテゴリ導線、一覧ページ、関連記事、パンくず相当導線を再設計。
- `characters.html` / `factions.html` / `history.html` の回遊性を改善。
- 内部リンク監査（孤立ページ 0 を目標）。

**成果物**
- `docs/navigation-map.md`
- リンク監査レポート

### Phase 4（Week 4）文体・文章品質の再構築
- 百科事典調の文体ガイド適用（断定、時制、用語統一）。
- 主要ページの段階的リライト（優先度順）。
- 参照表記ルールを統一（脚注の粒度、注記形式）。

**成果物**
- `docs/writing-style-ja-tenzu.md`
- リライト完了ページ群

### Phase 5（Week 5）メディア生成・差し替え
- 画像生成ガイドライン（構図、情報密度、著作権配慮）に基づき生成。
- 主要画像の段階置換（ヒーロー画像・組織ロゴ補助・地図図版等）。
- 画像キャプション/alt テキスト標準化。

**成果物**
- `docs/media-generation-guidelines.md`
- `assets/images/` 差し替えセット

### Phase 6（Week 6）総合QAと最終承認
- 回帰テスト（リンク切れ、レイアウト崩れ、可読性、表示速度）。
- 完了条件チェックシート採点。
- ユーザー最終レビュー＆微修正。

**成果物**
- `docs/final-qa-report.md`
- 完了判定ログ

## 4. スプリント単位の実行テンプレート（半自走）
各スプリントで以下を必須実行。
1. 前回レビュー反映確認
2. 対象タスク実装
3. セルフQA（差分確認）
4. スクリーンショット提出
5. ユーザーに詳細指示を依頼
6. タスク状態更新（`todo` / `in_progress` / `blocked` / `review` / `done`）

---

## 5. ユーザー確認ゲート（毎回）
毎回、次の4点をユーザーに確認する。
1. 見た目の寄せ方（厳密寄せ or 読みやすさ優先）
2. 文章トーン（厳密百科事典調の強度）
3. 画像生成のスタイル（写実/図解/混合）
4. 優先ページ（次スプリントの対象）

`needs_user_input=true` のタスクは、回答受領まで `blocked` を維持する。

---

## 6. 実行ループ（完了まで継続）
1. Backlog から次スプリント対象を選定
2. 実装・検証・自己監査
3. ユーザーに詳細確認依頼
4. タスク状態更新
5. 完了条件を再評価
6. 未達なら 1 に戻る

---

## 7. 直近2スプリントの具体アクション

### Sprint A（着手直後）
- P0-01: 現状UI監査を完了し、差分表を提出。
- P0-02: レイアウト骨格案を2案作成し、選択を依頼。
- P0-03: 色/文字/余白トークンの初版を提示。

### Sprint B（A承認後）
- P0-04: 記事テンプレート統一を1ページで試験導入。
- P1-02: infobox/hatnote/ambox/references の見た目調整。
- P2-01: 画像生成ガイドライン初版提出。
