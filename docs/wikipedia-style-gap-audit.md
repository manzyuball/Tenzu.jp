## 結論

現状のTenzu.jpは、Vector風UI、infobox、脚注、目次、hatnote、amboxなどの「見た目の部品」はかなりWikipediaに寄せている。一方で、文章の重心は実在のWikipediaよりも、作中組織の自己説明・世界観解説・批評的エッセイに近い。特に、出典が実在の検証可能な情報源として読者に確認できない点、導入部が定義より世界観の含意を優先する点、評価語と因果説明が多い点が、実在Wikipediaとの差を大きくしている。

## 参照した実在Wikipedia側の基準

1. **導入部は項目名・読み仮名・定義・記事全体の要約を置く。** 日本語版Wikipediaのスタイルマニュアルでは、冒頭文に項目名、読み仮名、定義を書き、その後に記事全体の要約を置くことが示されている。
2. **検証可能性が前提である。** Wikipediaの検証可能性方針は、信頼できる情報源から公表・出版された事実や視点を収録し、情報源を明記することを求める。
3. **中立的な観点は論争に参加しない。** 中立的な観点の方針では、重要な観点を比重に応じて公平に扱い、論争を「記述」することが重視されている。
4. **実記事の導入部は短い定義から始まる。** 「日本」は国名、地理的位置、面積、人口、首都、通称を短く列挙し、「楽天グループ」は本社所在地、事業領域、上場銘柄、グループ構造を先に示している。

## 代表記事 `faction-tenzu.html` の具体的な違和感

### 1. 導入部が長く、定義より解釈が早い

実在Wikipediaの企業記事は、法人名、読み、英名、本社所在地、事業内容、上場区分などから始まり、評価や論争は後段に回る。現状の天通記事は、1段落目で「情報複合体」と定義したあと、2〜5段落目で生活圏、公共圏、影響力、批判者の見方まで展開している。導入部として情報量は豊富だが、百科事典の冒頭というより、世界観の読み方を先に提示する文章になっている。

**改善例の方向性:**

```text
天通ホールディングス株式会社（てんつうホールディングス）は、大阪市北区に本社を置く西日本の情報通信会社。移動通信、固定回線、報道配信、電子商取引、広告配信、行政API接続などを手がける。通称は天通。
```

その後に「概要」節で、生活圏への浸透、三社体制、辞典運営を説明する方が実在Wikipediaに近い。

### 2. 「批評の語彙」が本文全体に分散している

「生活基盤としての粘着力が非常に高い」「危うさ」「自社正当化の装置」「何が公的知識に見えるかを編集する会社」などは、読み物としては魅力的だが、Wikipedia本文では出典付きの第三者評価として扱うべき表現である。実在Wikipedia風にするなら、本文中の評価語を以下の形に分解する必要がある。

- 誰がそう評価したのか。
- いつの資料・報道・研究なのか。
- その評価に反対・留保する立場はあるのか。
- 記事の地の文で断定する必要があるのか。

### 3. 出典が「確認できる資料」ではなく「設定の小道具」になっている

出典区分そのものは良い設計だが、現状では読者が現実に確認できるURL、出版社、発行年、ページ番号、閲覧日がない。これは作中資料網としては自然だが、実在Wikipediaの文章らしさからは離れる。実在Wikipedia風に寄せるなら、架空資料でも書誌情報の粒度を上げるとよい。

```text
天通社史編纂室『回線から公共圏へ』天通出版、2024年、pp. 112-118。
```

のように、発行者・年・ページ範囲を付けると、少なくとも「Wikipediaに掲載されそうな出典表記」に近づく。

### 4. 見出しが設定解説向きである

「経済・社会的位置」「対外関係」は世界設定を語るには便利だが、企業記事としてはやや抽象的である。実在Wikipedia風にするなら、企業情報の分類に寄せる。

推奨構成:

1. 概要
2. 沿革
3. 事業
4. 経営・組織
5. 関係会社・資本関係
6. サービス
7. 社会的影響
8. 批判・不祥事・論争
9. 脚注
10. 関連項目
11. 外部リンク

### 5. 世界観情報が「読者に先回りして解釈を教える」

現状の記事は、天通が何を意味する存在かをかなり丁寧に説明している。そのため、設定資料としては読みやすい。一方、実在Wikipediaは一般に、読者が資料を読んで判断できるよう、評価や意味づけの押し出しを抑える。Wikipedia風を強めるほど、文章の面白さは下がるが、百科事典らしさは上がる。

## トップページ `index.html` の差分

`index.html`は「天通が運営する公開資料網」であること、常時改訂型であること、南北双方の一次情報を照合することを明示している。これは作中サイトの設定として非常に分かりやすいが、実在Wikipediaのトップページというより、企業運営の公開データベースの説明に近い。

Wikipedia風トップページに寄せる場合は、以下を増やすとよい。

- 「主要項目」よりも「分野別入口」「新着記事」「特集記事」「今日は何の日」風の面を作る。
- 「編集方針」はトップ本文ではなく、別ページまたはフッター導線へ移す。
- 「本サイトは天通が運営」という作中性は残しつつ、百科事典の入口としての情報密度を上げる。

## 画像・写真・図を追加する場合のポイント

実在Wikipedia風に寄せるうえで、画像は「見栄えを良くする装飾」ではなく、本文で説明している事実を読者が確認しやすくする補助資料として扱う必要がある。Tenzu.jpでは架空世界の画像を生成・配置するため、実在Wikipediaと完全に同じ検証可能性は実現できないが、以下の運用を入れるとWikipediaらしさが増す。

### 1. 画像の役割を先に決める

画像を追加する前に、その画像がどの情報を補うのかを決める。記事ごとに、次の優先順位で配置するとよい。

1. **識別画像:** ロゴ、旗、紋章、本社外観、代表的施設など、主題を識別する画像。infobox内に1〜2点まで置く。
2. **位置・範囲を示す図:** 地図、勢力圏、交通網、通信網、停戦線など、文章だけでは空間関係が分かりにくい情報を示す図。
3. **構造を示す図:** 資本関係、組織図、部門間データ連携、年表など、複雑な関係を整理する図。
4. **本文補助の写真:** 会議場、街頭、基地局、物流拠点、庁舎など、本文中の具体的な節に対応する写真。

雰囲気重視のイメージ画像、映画ポスター的なキービジュアル、説明対象が本文にない画像は避ける。Wikipedia風にするなら、画像は「かっこいいから置く」のではなく「この記事の事実確認や理解に必要だから置く」という順序にする。

### 2. キャプションは短く、説明対象を明示する

キャプションは、画像が何を示しているかを短く説明する。本文の主張をキャプションで補強したり、画像だけで新情報を語りすぎたりしない。

良い例:

```text
大阪市北区に置かれる天通本社ビル（2026年作中資料）
```

避けたい例:

```text
西日本の世論を支配する巨大情報企業・天通の威容を示す本社ビル
```

後者は評価語が多く、百科事典のキャプションというより宣伝・批評の文に見える。キャプションでは「いつ」「どこ」「何を写した/示したか」を優先し、解釈は本文の適切な節へ移す。

### 3. 代替テキストをキャプションとは別に考える

Wikipediaの画像表示ヘルプでは、画像にはキャプションと代替文を設定できる。Tenzu.jpでも、`alt`はキャプションのコピーではなく、画像が見えない読者に必要な情報を伝える文にする。

例:

```html
<img src="assets/images/tentsu_hq_building_v2.png" alt="高層棟と低層棟で構成される天通本社ビルの外観">
<small>大阪市北区に置かれる天通本社ビル</small>
```

キャプションは画面上に表示される説明、代替テキストは画像の内容を置き換える説明として分ける。ロゴ画像の`alt`は「天通ロゴ」、地図なら「京都政権と東京政権の支配範囲を示す地図」のようにする。

### 4. 画像ごとに「作中出典」と「現実の制作情報」を持たせる

実在Wikipediaでは、ファイルページに出典、著作権表示、ライセンス表示を置く。Tenzu.jpは静的サイトなので、同じ機能をそのまま持てないが、画像台帳を作ることで近い運用にできる。

推奨する画像台帳の項目:

| 項目 | 内容 | 例 |
| --- | --- | --- |
| ファイル名 | `assets/images/`以下の実ファイル名 | `tentsu_hq_building_v2.png` |
| 記事内用途 | infobox、沿革節、地図、組織図など | infobox本社画像 |
| 作中出典 | 作中での撮影者、機関、資料名 | 天通広報局資料写真、2026年 |
| 現実の制作情報 | 生成画像、手描き、撮影、編集元など | AI生成後に手動補正 |
| 権利メモ | 自作、生成、外部素材なし、参照元あり等 | リポジトリ内自作素材 |
| 置換条件 | 後で差し替える条件 | 地図設定確定後に再生成 |

この台帳は、将来的には `docs/media-generation-guidelines.md` または `assets/images/README.md` として独立させるとよい。

### 5. 生成画像は「実在写真風」より「作中資料写真」として扱う

AI生成画像を使う場合、実在の報道写真や実在企業の写真に見えすぎると、Wikipedia風ではなく偽資料風になる。Tenzu.jpでは、生成画像を置く際に以下を守るとよい。

- 実在企業、実在ロゴ、実在人物、実在報道写真に似せない。
- キャプションや画像台帳で「作中資料」「生成画像」「模式図」を区別する。
- 人物写真は、本人確認や肖像権を連想させるため、架空人物でも使用目的を絞る。
- 事件、災害、戦闘、抗議活動などの画像は、読者に実在ニュース写真と誤認されやすいので、写真より地図・模式図・年表を優先する。

特に南北日本戦記のような政治・戦争・企業支配を扱う設定では、リアルな写真を多用すると「架空の百科事典」ではなく「偽ニュース資料」に見える危険がある。Wikipedia風にするなら、写真のリアリティよりも、キャプションと出典管理の慎重さを優先する。

### 6. 図・地図は本文の論点を整理するために使う

架空世界の記事では、地図、組織図、年表、資本関係図の効果が大きい。写真よりも図の方が、Wikipedia的な資料性を出しやすい場合が多い。

追加候補:

- **勢力記事:** 支配地域図、主要拠点図、関係勢力図。
- **企業記事:** 資本関係図、事業別売上構成図、主要サービス年表。
- **人物記事:** 略歴年表、所属組織の変遷図。
- **歴史記事:** 年表、停戦線の変化、主要事件の位置図。

図は本文中の数字・関係と一致させる。本文にない数値や関係を図だけで出すと、検証できない独自情報が増えたように見えるため、図で示す情報にも脚注や作中資料名を付ける。

### 7. 配置と枚数は抑制する

Wikipediaの画像利用方針でも、過剰な写真配置は読者の閲覧を妨げる可能性がある。Tenzu.jpでも、1記事あたりの画像枚数は記事の長さに応じて抑える。

目安:

- 短い記事: infobox画像1点のみ。
- 標準記事: infobox画像1点 + 本文補助1〜2点。
- 長大記事: infobox画像1〜2点 + 地図/図表2〜4点 + 必要ならギャラリー。

画像が多い場合は、本文の途中に無理に差し込むより、「ギャラリー」節を作る。ただし、ギャラリーは本文理解に必要な画像だけに限定し、世界観の画集にしない。

### 8. ファイル形式と命名を統一する

実在Wikipediaの画像利用方針では、写真はJPEG、アイコン・図形・地図・旗などはSVGまたはPNGが推奨されている。Tenzu.jpでも以下を標準にすると管理しやすい。

- 写真風画像: `.jpg` または `.png`。生成画像で透過不要なら `.jpg` も検討する。
- ロゴ・旗・紋章: 可能なら `.svg`、難しければ透過 `.png`。
- 地図・組織図・模式図: 編集しやすい `.svg` を優先し、表示確認用に `.png` を併用してもよい。
- ファイル名: `subject_kind_version.ext` のように、記事名・種類・版を入れる。例: `tenzu_hq_2026_v2.png`、`north_south_japan_ceasefire_map_v1.svg`。

### 9. 記事への組み込み例

現行の `faction-tenzu.html` では、infobox用にロゴと本社ビル画像を配置している。Wikipedia風を高めるなら、画像HTMLは次の観点で整える。

```html
<img src="assets/images/tentsu_logo.png" width="150" alt="天通のロゴ">
<br>
<small>天通のロゴ</small>
<br><br>
<img src="assets/images/tentsu_hq_building_v2.png" width="220" alt="大阪市北区に置かれる天通本社ビルの外観">
<br>
<small>大阪市北区の天通本社ビル（作中資料、2026年）</small>
```

さらに、画像台帳側に「作中出典」「現実の制作情報」「権利メモ」を書けば、本文・画像・管理資料の3点がつながり、実在Wikipediaのファイルページに近い運用になる。

## 優先改善タスク

1. **導入部リライト基準を作る。** すべての記事冒頭を「項目名・読み・英名/別名・定義・基本属性」に揃える。
2. **評価語の棚卸しをする。** 「危うい」「象徴」「装置」「非常に高い」などを抽出し、第三者評価・批判節・削除候補に分類する。
3. **作中出典の書誌粒度を上げる。** 出版者、年、ページ、資料種別、URL風IDを持たせる。
4. **企業記事テンプレートを再定義する。** `faction-tenzu.html`を試験対象に、見出し順を実在企業記事寄りへ変更する。
5. **トップページの役割を決める。** 「作中企業資料網」らしさを優先するか、「百科事典メインページ」らしさを優先するかを明文化する。
6. **画像・図版運用を標準化する。** 画像台帳、キャプション規約、代替テキスト、生成画像の作中出典表示を整える。
7. **用語リンク網を増やす。** 公共圏、行政API、推薦基盤、三社体制などの用語を記事化または用語集化する。

## まとめ

実在Wikipediaとの最大差は、CSSや部品ではなく、文章の役割にある。Tenzu.jpの文章は「世界観を読ませる」ために、評価、含意、構造説明を前面に出している。実在Wikipediaの文章は「検証可能な事実を整理する」ために、定義、属性、沿革、出典、論争を分離する。したがって、Wikipediaらしさを上げるには、見た目の改修よりも、導入部・見出し・出典・評価語・画像運用の統制が最優先である。
